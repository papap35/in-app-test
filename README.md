# 📱 In-App Browser 儲存機制與導流行為驗證測試 (PoC)

本專案為一個輕量化的前端驗證網頁 (Proof of Concept)，旨在測試與實機驗證在主流社群 App (LINE、FB、IG) 的內建瀏覽器 (In-App Browser) 環境中，Web Storage (`LocalStorage` / `SessionStorage`) 的運作行為、多頁面獨立計時可行性，以及跨瀏覽器資料隔離的技術限制。

## 🌐 專案動態展示 (Live Demo)

* **POC 測試控制台**：[https://papap35.github.io/in-app-test/index.html](https://papap35.github.io/in-app-test/index.html)

---

## 🔍 核心驗證三大問題

1. **獨立計時能力**：在 In-App 瀏覽器中，不同的直播間 (不同 URL/頁面 ID) 能否各自用獨立的 Key 紀錄倒數秒數而不互相覆蓋？
2. **生命週期差異**：在 In-App 瀏覽器中，`SessionStorage`（分頁級生命週期）與 `LocalStorage`（持久化生命週期）何者更適合用來保底計時？
3. **資料隔離邊界**：當使用者從社群內建瀏覽器點擊外跳至原生瀏覽器 (Safari / Chrome) 時，資料能否共用？秒數能否接續？

---

## 📂 檔案結構說明

專案由兩個極簡的 HTML 靜態網頁組成，可直接託管於任何支援 HTTPS 的免費空間（如 GitHub Pages、Vercel 或 Ngrok）：

* `index.html`：直播間【第一頁】測試環境，綁定頁面 ID 為 `page-1`。
* `page2.html`：直播間【第二頁】測試環境，綁定頁面 ID 為 `page-2`。

## 🛠️ 實機測試步驟與預期結果

請將本專案部署至公開的 HTTPS 網址，並將 `index.html` 的連結發送至 **LINE 聊天室** 或貼在 **FB/IG** 上進行以下實驗：

### 實驗一：驗證「多頁面獨立紀錄」與「切換延續性」
1. 在 LINE 中點開 `index.html`，看著畫面從 `⏳ 儲存空間讀取中...` 變成 `Local: 10s | Session: 10s` 並開始倒數。
2. 讓它倒數至 **6 秒**，點擊 **「👉 前往第二頁 (page-2)」**。
3. **觀察點 A**：第二頁網頁載入後，會重新從 **10 秒** 開始倒數。
    * *技術結論*：**驗證成功。** 只要給定獨立的 Page ID Key，不同直播間的秒數在同一個 WebView 中不會互相覆蓋。
4. 在第二頁讓它倒數至 **4 秒**，點擊 **「👈 返回第一頁 (page-1)」**。
5. **觀察點 B**：第一頁網頁載入後，秒數不會重置回 10 秒，而是**精準接續剛剛離開時的 5~6 秒**繼續往下數。
    * *技術結論*：**驗證成功。** 在未關閉瀏覽器的情況下進行內部頁面跳轉，`SessionStorage` 與 `LocalStorage` 都能完美留存當前分頁的狀態。

### 實驗二：驗證「內建瀏覽器關閉 (點擊 X) 的生命週期」
1. 在 LINE 網頁中，讓任何一頁倒數到一半。
2. 點擊內建瀏覽器左上角或右上角的 **「X」直接關閉網頁**，返回聊天室。
3. 再次點擊連結進入網頁。
4. **觀察點**：畫面閃過讀取中之後，你會發現：
    * `Local` 的秒數：**繼續剛剛未數完的秒數。**
    * `Session` 的秒數：**被徹底清空，重新從 10 秒開始倒數。**
    * *技術結論*：在 In-App 瀏覽器中，點擊「X」會銷毀當前 WebView 的分頁實例，導致 `SessionStorage` 蒸發。因此**本功能若要實作，前端必須統一採用 `LocalStorage`**。

### 實驗三：驗證「外跳原生瀏覽器 (Safari / Chrome) 的隔離限制」
1. 在 LINE 內建瀏覽器中，讓網頁倒數到 **5 秒**。
2. 點擊 LINE 右上角的 `...` 符號，選擇 **「使用 Safari 開啟」** (iOS) 或 **「在 Chrome 中開啟」** (Android)。
3. **觀察點**：新開的原生瀏覽器中，畫面會顯示 `Local: 10s | Session: 10s` 重新開始計時。
    * *技術結論*：**完全隔離。** 基於作業系統安全沙盒機制，社群 App 的內建 WebView 儲存空間與系統獨立瀏覽器是 100% 斷開的，數據無法跨 App 傳遞。

## 💡 最終技術決策建議 (給團隊與 PM/Designer)

基於上述 PoC 實測結果，針對「手機網頁導流下載安裝與註冊 App」功能，給予以下開發與規格建議：

1.  **放棄跨環境秒數接續**：由於沙盒機制限制，使用者一旦外跳瀏覽器，秒數必然會重新計算（無法從第 4 秒接續）。這屬於系統底層限制，非前端程式碼可繞過。
2.  **計時器數據全面採用 LocalStorage**：為防範使用者不小心滑掉網頁、重複進出直播間，前端應摒棄 `SessionStorage`，統一使用帶有直播間 ID 的 `LocalStorage` 作為唯一的秒數紀錄媒介（例如：`local_countdown_page-1`）。
3.  **分流優化導流體驗 (最關鍵建議)**：
    * **原生瀏覽器 (Safari/Chrome)**：使用者體驗好、無下載阻擋，可安心維持原設計「倒數 X 秒後強制轉導」。
    * **社群內建瀏覽器 (LINE/FB/IG)**：考慮到下載連結極易被社群直接阻擋（即便秒數到了也無法跳轉 App Store），且資料無法與外開瀏覽器共用，**建議在此環境下直接取消 X 秒倒數**。改為「一偵測到 In-App 環境，立即顯示全螢幕引導蓋板，指導使用者點擊右上角外跳至外部瀏覽器」，這樣能帶來最高的漏斗轉化率。
